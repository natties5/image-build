# ประวัติการ Commit และกลยุทธ์การจัดการ Branch (Commit History and Branching Strategy)

โปรเจกต์นี้ใช้แนวทางการจัดการข้อความ Commit และการตั้งชื่อ Branch ที่เป็นระบบ เพื่อรักษาประวัติการทำงานให้สะอาดและเข้าใจง่าย

## คำแนะนำการตั้งชื่อ Branch (Branch Naming Guidance)

การตั้งชื่อ Branch ควรระบุวัตถุประสงค์ให้ชัดเจน:

- `codex/<topic>`: Branch หลักสำหรับการพัฒนาฟีเจอร์ใหม่ หรือการปรับปรุงโครงสร้างครั้งใหญ่
- `feat/<topic>`: ฟีเจอร์ขนาดเล็กถึงขนาดกลาง
- `fix/<topic>`: การแก้ไขข้อผิดพลาด (Bug fixes)
- `refactor/<topic>`: การจัดระเบียบโค้ดหรือล้างโค้ดใหม่ โดยไม่มีการเปลี่ยนพฤติกรรมเดิม
- `docs/<topic>`: การปรับปรุงเอกสาร

ตัวอย่าง: `codex/add-multi-os-auto-download-support`

## รูปแบบข้อความ Commit แบบมาตรฐาน (Conventional Commit Style)

เราส่งเสริมการใช้ข้อความ Commit แบบมาตรฐานเพื่อความชัดเจน:

- `feat: ...`: เพิ่มฟีเจอร์ใหม่
- `fix: ...`: แก้ไขข้อผิดพลาด
- `refactor: ...`: จัดระเบียบหรือล้างโค้ดใหม่
- `docs: ...`: ปรับปรุงเอกสาร
- `chore: ...`: งานทั่วไป (เช่น อัปเดต dependencies, ปรับปรุง CI)

ตัวอย่าง: `refactor: organize config layers docs and history without changing pipeline behavior`

## การติดตามการเปลี่ยนแปลง (Change Tracking)

การตัดสินใจด้านสถาปัตยกรรมที่สำคัญและการเปลี่ยนแปลงครั้งใหญ่ควรถูกบันทึกไว้

### 1. บันทึกการตัดสินใจทางสถาปัตยกรรม (Architectural Decision Records - ADR)
เก็บไว้ในโฟลเดอร์ `doc/adr/` โดยแต่ละไฟล์ควรมีหมายเลขและวันที่กำกับ เพื่อระบุ:
- บริบทของการตัดสินใจ
- ทางเลือกอื่นๆ ที่พิจารณา
- วิธีแก้ปัญหาที่เลือกและเหตุผลประกอบ

### 2. ประวัติการเปลี่ยนแปลงครั้งสำคัญ
สำหรับการปรับปรุงโครงสร้างหรือฟีเจอร์หลักแต่ละครั้ง ควรมีเอกสารสรุปในโฟลเดอร์ `doc/`
ตัวอย่าง: `doc/summary-config-reorg-th.md`

## สรุปการเปลี่ยนแปลงล่าสุด
เพื่อทำความเข้าใจว่ามีอะไรเปลี่ยนไปบ้างในระยะหลัง นักพัฒนาหรือผู้ดูแลระบบควรตรวจสอบจาก:
1. `git log --oneline --graph`
2. ไฟล์สรุป (Summary files) ที่เกี่ยวข้องในโฟลเดอร์ `doc/`
3. ADR ต่างๆ ในโฟลเดอร์ `doc/adr/`
